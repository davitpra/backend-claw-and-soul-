# ClawAndSoul — Plan de Implementación: Vinculación de Formatos con Shopify

## Objetivo

Conectar la tabla `formats` con las variantes reales de Shopify para que los tamaños, precios y disponibilidad siempre estén sincronizados. El frontend usa la Storefront API para datos comerciales (precios, stock) y el backend mantiene las dimensiones de generación.

---

## Fase 1 — Cambios en el Schema

### 1.1 Modificar `product_references`

Agregar `shopify_handle` para queries en la Storefront API:

```prisma
model ProductReference {
  id                String    @id @default(uuid())
  shopifyProductId  String    @unique @map("shopify_product_id")
  shopifyHandle     String    @map("shopify_handle")          // NUEVO
  name              String
  displayName       String    @map("display_name")
  description       String?   @db.Text
  isActive          Boolean   @default(true) @map("is_active")
  createdAt         DateTime  @default(now()) @map("created_at")
  updatedAt         DateTime  @updatedAt @map("updated_at")

  compatibilities StyleFormatProductCompat[]
  generations     Generation[]

  @@map("product_references")
}
```

### 1.2 Modificar `formats`

Agregar `shopify_variant_option` para mapear al valor de la opción "Size" en Shopify:

```prisma
model Format {
  id                    String    @id @default(uuid())
  name                  String    @unique
  displayName           String    @map("display_name")
  aspectRatio           String    @map("aspect_ratio")
  width                 Int
  height                Int
  shopifyVariantOption  String    @map("shopify_variant_option")  // NUEVO: "8x10", "16x20"
  isActive              Boolean   @default(true) @map("is_active")
  createdAt             DateTime  @default(now()) @map("created_at")
  updatedAt             DateTime  @updatedAt @map("updated_at")

  compatibilities StyleFormatProductCompat[]
  generations     Generation[]

  @@map("formats")
}
```

### 1.3 Crear nueva tabla `product_format_variants`

Esta tabla vincula cada producto + formato con su variante exacta de Shopify. Es la pieza que conecta tu sistema de generación con el catálogo comercial:

```prisma
model ProductFormatVariant {
  id                 String    @id @default(uuid())
  productRefId       String    @map("product_ref_id")
  formatId           String    @map("format_id")
  shopifyVariantId   String    @map("shopify_variant_id")    // ID de la variante en Shopify
  shopifyVariantTitle String   @map("shopify_variant_title") // "8x10 / Matte"
  isActive           Boolean   @default(true) @map("is_active")
  createdAt          DateTime  @default(now()) @map("created_at")
  updatedAt          DateTime  @updatedAt @map("updated_at")

  productRef  ProductReference @relation(fields: [productRefId], references: [id])
  format      Format           @relation(fields: [formatId], references: [id])

  @@unique([productRefId, formatId])
  @@index([shopifyVariantId])
  @@map("product_format_variants")
}
```

### 1.4 Migración en dos pasos

Dado que `product_references` ya tiene datos en producción, las columnas nuevas se agregan primero como nullable, se poblan, y luego se hacen obligatorias:

## Fase 1.5 — Migración de Datos Existentes

Esta fase se ejecuta una vez para poblar los campos nuevos y vincular variantes. Se realiza antes de hacer las columnas NOT NULL.

### 1.5.1 Backfill de `shopify_handle` en `product_references`

Consulta la Shopify Admin API para obtener el handle de cada producto existente:

```typescript
// scripts/backfill-handles.ts

async function backfillHandles() {
  const products = await prisma.productReference.findMany({
    where: { shopifyHandle: null }
  });

  console.log(`📦 ${products.length} productos sin shopify_handle`);

  for (const product of products) {
    try {
      const shopifyProduct = await shopifyAdmin.get(
        `/products/${product.shopifyProductId}.json`
      );

      await prisma.productReference.update({
        where: { id: product.id },
        data: { shopifyHandle: shopifyProduct.product.handle }
      });

      console.log(`✅ ${product.displayName} → handle: ${shopifyProduct.product.handle}`);
    } catch (error) {
      console.error(`❌ Error en ${product.displayName}: ${error.message}`);
    }
  }
}
```

### 1.5.2 Configurar `shopify_variant_option` en formatos existentes

Si ya tienes formatos, hay que asignarles el valor de la opción "Size" que corresponde en Shopify. Esto se hace manualmente o con un script si los nombres ya coinciden:

```typescript
// scripts/backfill-format-options.ts

// Si los nombres de tus formatos ya contienen el tamaño, puedes mapear automáticamente:
const FORMAT_TO_OPTION: Record<string, string> = {
  'square_8x8':      '8x8',
  'portrait_8x10':   '8x10',
  'landscape_16x20': '16x20',
  'portrait_24x36':  '24x36',
  // ... agregar todos los formatos existentes
};

async function backfillFormatOptions() {
  for (const [formatName, optionValue] of Object.entries(FORMAT_TO_OPTION)) {
    const result = await prisma.format.updateMany({
      where: { name: formatName, shopifyVariantOption: null },
      data: { shopifyVariantOption: optionValue }
    });

    if (result.count > 0) {
      console.log(`✅ ${formatName} → shopifyVariantOption: ${optionValue}`);
    }
  }

  // Verificar si quedaron formatos sin configurar
  const missing = await prisma.format.findMany({
    where: { shopifyVariantOption: null }
  });

  if (missing.length > 0) {
    console.warn(`⚠️ ${missing.length} formatos sin shopifyVariantOption:`);
    missing.forEach(f => console.warn(`   - ${f.name} (${f.displayName})`));
  }
}
```

### 1.5.3 Poblar `product_format_variants` desde Shopify

Recorre los productos existentes, trae sus variantes de Shopify, y crea los vínculos con los formatos internos:

```typescript
// scripts/seed-product-format-variants.ts

async function seedProductFormatVariants() {
  const products = await prisma.productReference.findMany({
    where: { isActive: true }
  });

  let created = 0;
  let warnings = 0;

  for (const product of products) {
    try {
      const shopifyProduct = await shopifyAdmin.get(
        `/products/${product.shopifyProductId}.json`
      );

      for (const variant of shopifyProduct.product.variants) {
        // Extraer la opción "Size" (asumiendo que es option1)
        const sizeOption = variant.option1;

        const format = await prisma.format.findFirst({
          where: { shopifyVariantOption: sizeOption }
        });

        if (format) {
          await prisma.productFormatVariant.upsert({
            where: {
              productRefId_formatId: {
                productRefId: product.id,
                formatId: format.id
              }
            },
            create: {
              productRefId: product.id,
              formatId: format.id,
              shopifyVariantId: String(variant.id),
              shopifyVariantTitle: variant.title,
            },
            update: {
              shopifyVariantId: String(variant.id),
              shopifyVariantTitle: variant.title,
            }
          });
          created++;
          console.log(`✅ ${product.displayName} + ${format.displayName} → variant ${variant.id}`);
        } else {
          warnings++;
          console.warn(
            `⚠️ ${product.displayName}: variante "${sizeOption}" sin formato configurado`
          );
        }
      }
    } catch (error) {
      console.error(`❌ Error en ${product.displayName}: ${error.message}`);
    }
  }

  console.log(`\n📊 Resultado: ${created} vínculos creados, ${warnings} warnings`);
}
```

### 1.5.4 Validar compatibilidades existentes

Si ya tienes reglas en `style_format_product_compat`, verificar que tengan variante vinculada:

```typescript
// scripts/validate-compat.ts

async function validateCompat() {
  const rules = await prisma.styleFormatProductCompat.findMany({
    where: { isActive: true },
    include: {
      format: true,
      productRef: true
    }
  });

  let valid = 0;
  let broken = 0;

  for (const rule of rules) {
    const hasVariant = await prisma.productFormatVariant.findFirst({
      where: {
        productRefId: rule.productRefId,
        formatId: rule.formatId,
        isActive: true
      }
    });

    if (hasVariant) {
      valid++;
    } else {
      broken++;
      console.warn(
        `⚠️ Regla ${rule.id}: ${rule.productRef.displayName} + ${rule.format.displayName}`
        + ` → sin variante vinculada (no se mostrará al usuario)`
      );
    }
  }

  console.log(`\n📊 Compatibilidades: ${valid} válidas, ${broken} sin variante`);
}
```

## Fase 2 — Actualizar Sincronización de Shopify

### 2.1 Modificar lógica de upsert de productos

Actualizar `ProductSyncService` para incluir `shopify_handle` en el mapeo:

```
Shopify Product              →    product_references
─────────────────────────────────────────────────────
product.id                   →    shopify_product_id
product.handle               →    shopify_handle        // NUEVO
product.handle               →    name
product.title                →    display_name
product.body_html (stripped) →    description
product.status === 'active'  →    is_active
```

### 2.2 Sincronizar variantes

Extender el sync para que al procesar cada producto, también sincronice sus variantes contra `product_format_variants`:

```
Para cada producto de Shopify:
  1. Upsert en product_references (existente)
  2. Obtener variantes del producto
  3. Para cada variante:
     a. Extraer la opción "Size" (ej: "8x10")
     b. Buscar el format que tenga shopify_variant_option === "8x10"
     c. Si existe el format:
        - Upsert en product_format_variants:
          { productRefId, formatId, shopifyVariantId, shopifyVariantTitle }
     d. Si no existe el format:
        - Log warning: "Variante '8x10' de producto 'Poster' no tiene formato configurado"
        - Continuar (no bloquear el sync)
```

### 2.3 Actualizar webhooks

Los webhooks de `products/update` ya incluyen variantes en el payload. Extender el handler para que procese variantes igual que el cron:

```
POST /webhooks/shopify/product/update
  1. Verificar HMAC
  2. Responder 200
  3. Encolar job 'shopify-sync' con payload completo
  4. Worker:
     a. Upsert producto en product_references
     b. Sync variantes → product_format_variants
     c. Desactivar variantes que ya no existen en Shopify
     d. Registrar en sync_logs
```

**Tareas:**
- [ ] Modificar `ProductSyncService.upsertProduct()` para incluir `shopifyHandle`
- [ ] Crear `ProductSyncService.syncVariants()` para sincronizar variantes
- [ ] Actualizar webhook handler para procesar variantes
- [ ] Agregar logs/warnings para variantes sin formato configurado

### 3.2 Query interno

```
SELECT f.*, pfv.shopify_variant_id, pfv.shopify_variant_title
FROM style_format_product_compat sfpc
JOIN formats f ON f.id = sfpc.format_id
JOIN product_format_variants pfv ON pfv.product_ref_id = sfpc.product_ref_id
                                 AND pfv.format_id = sfpc.format_id
WHERE sfpc.product_ref_id = :product_id
  AND sfpc.is_active = true
  AND f.is_active = true
  AND pfv.is_active = true
GROUP BY f.id, pfv.shopify_variant_id, pfv.shopify_variant_title
```

## Fase 4 — Integración Frontend con Storefront API

### 4.1 Query de Storefront API

El frontend consulta precio y disponibilidad usando el handle del producto:

```graphql
query GetProductByHandle($handle: String!) {
  product(handle: $handle) {
    id
    title
    variants(first: 50) {
      edges {
        node {
          id                    # gid://shopify/ProductVariant/12345
          title                 # "8x10 / Matte"
          availableForSale
          price {
            amount
            currencyCode
          }
          selectedOptions {
            name                # "Size"
            value               # "8x10"
          }
        }
      }
    }
  }
}
```

### 4.2 Flujo de cruce en el frontend

```
1. Usuario elige producto
   → Frontend obtiene shopify_handle de tu API: GET /products

2. Frontend hace dos llamadas en paralelo:
   a. Storefront API → variantes con precios y stock
   b. Tu API → GET /compat/formats?product_id=X → formatos compatibles con shopifyVariantId

3. Frontend cruza ambas respuestas por shopifyVariantId:
   - Para cada formato compatible, busca la variante de Shopify con el mismo ID
   - Resultado: lista de formatos con dimensiones de generación + precio + disponibilidad

4. Muestra al usuario:
   "8x10 Retrato — $29.99 — Disponible"
   "16x20 Retrato — $49.99 — Agotado"

5. Formatos sin variante activa en Shopify → no se muestran
   (aunque exista la compatibilidad, si Shopify dice agotado, no se ofrece)
```

### 4.3 Estructura de datos en el frontend

```typescript
interface FormatOption {
  // De tu API (backend)
  formatId: string;
  name: string;
  displayName: string;
  aspectRatio: string;
  width: number;           // dimensiones para generación
  height: number;

  // De Storefront API (Shopify)
  shopifyVariantId: string;
  price: string;
  currencyCode: string;
  availableForSale: boolean;
}

// Función de cruce
function mergeFormatsWithVariants(
  compatFormats: CompatFormat[],       // de tu API
  shopifyVariants: ShopifyVariant[]    // de Storefront API
): FormatOption[] {
  return compatFormats
    .map(format => {
      const variant = shopifyVariants.find(
        v => v.id === format.shopifyVariantId
      );
      if (!variant) return null;

      return {
        formatId: format.id,
        name: format.name,
        displayName: format.displayName,
        aspectRatio: format.aspectRatio,
        width: format.width,
        height: format.height,
        shopifyVariantId: variant.id,
        price: variant.price.amount,
        currencyCode: variant.price.currencyCode,
        availableForSale: variant.availableForSale,
      };
    })
    .filter(Boolean)
    .filter(f => f.availableForSale);  // solo mostrar disponibles
}
```

Cuando el usuario termina el flujo y quiere comprar, el frontend crea un checkout en Shopify con el `variantId` exacto:

```graphql

  checkoutCreate(input: {
    lineItems: [{
      variantId: $variantId
      quantity: 1
```

## Resumen del Flujo Completo

```
SETUP (admin, una vez):
  1. Migración de schema (columnas nullable + tabla product_format_variants)
  2. Backfill: poblar shopify_handle en productos existentes
  3. Configurar shopifyVariantOption en formatos existentes
  4. Seed: vincular product_format_variants desde Shopify
  5. Validar compatibilidades existentes
  6. Migración final: hacer columnas NOT NULL
  7. Configurar matriz de compatibilidad si faltan reglas

RUNTIME (usuario):
  1. Login
  2. Elige producto → GET /products
  3. Frontend en paralelo:
     a. GET /compat/formats?product_id=X → formatos + shopifyVariantId
     b. Storefront API → variantes con precios y stock
  4. Cruce: muestra formatos con precio y disponibilidad
  5. Elige formato
  6. Crea mascota + sube fotos
  7. GET /compat/styles?product_id=X&format_id=Y → estilos compatibles
  8. Elige estilo
  9. POST /generations → encola en BullMQ
  10. Polling GET /generations/:id/status
  11. Resultado listo
  12. checkoutCreate con variantId + generation_id como custom attribute
  13. Redirige a Shopify checkout → pago
  14. Webhook orders/paid → asociar orden con generación
```

## Riesgos y Mitigaciones

| Riesgo | Impacto | Mitigación |
|--------|---------|------------|
| Datos existentes sin shopify_handle | Migración final falla si quedan nulls | Backfill script + verificación antes de NOT NULL |
| Formatos existentes sin shopifyVariantOption | Variantes no se vinculan a formatos | Mapeo manual en backfill-format-options.ts, validar antes de continuar |
| Shopify cambia el nombre de la opción "Size" | Formatos no se vinculan | Admin configura el nombre del option en product_references |
| Variante nueva en Shopify sin formato configurado | No se muestra al usuario | Warning en sync + listado en admin dashboard |
| Storefront API rate limits | Frontend no puede mostrar precios | Cache de variantes en frontend (TTL 5 min) |
| Variante agotada durante el flujo de generación | Usuario genera pero no puede comprar | Validar disponibilidad antes de POST /generations |
